# 정책명 직접 매칭 성능 비교

측정일: 2026-07-16

## 비교 대상

| 버전 | 검색 경로 |
| --- | --- |
| 기준선 | `a2d0205` LangGraph 적용 후 버전. 정책명을 별도로 해석하지 않고 전역 벡터 검색을 수행한다. |
| 개선안 | 정책명 직접 매칭 후 단일 정책이면 해당 정책의 부모 문서를 조회하고, 모호하거나 미일치일 때만 기존 경로를 사용한다. |

## 측정 방법

- 실제 PostgreSQL의 활성 정책 467건을 사용했다.
- 상세 문서와 검색 청크가 모두 있는 정책 중 직접 매칭 가능한 5건을 자동 추출했다.
- 질의 형식은 `<정책 공식 제목> 신청 기간 알려줘`다.
- 기준선은 local Ollama `bge-m3` 임베딩으로 `retrieve_policy_chunk_sources`를 호출했다.
- 개선안은 캐시가 준비된 상태에서 `정책명 매칭 → retrieve_policy_document_sources`를 호출했다.
- 정답은 정책 선택을 다시 요구하지 않고, 대상 정책 ID의 근거만 반환한 경우로 정의했다.
- LLM 답변 생성 시간은 두 경로 모두의 공통 후속 단계이므로 제외했다.

## 해석과 한계

- 성능 개선의 원인은 LangGraph가 아니라 정책명이 명시된 경우 임베딩·전역 벡터 검색을 건너뛰는 직접 매칭이다.
- 일반 질문, 정책명이 없는 질문, 오타가 있는 질문은 기존 벡터 검색으로 내려가므로 이 수치가 적용되지 않는다.
- 별칭이 여러 정책에 동시에 걸리면 임의 선택하지 않고 기존 정책 선택 흐름으로 보낸다.
- 현재 측정은 local 임베딩 기준이다. cloud 임베딩에서도 직접 매칭은 임베딩 호출을 생략하므로 비용·지연 감소 효과가 예상되지만, 별도 측정이 필요하다.
